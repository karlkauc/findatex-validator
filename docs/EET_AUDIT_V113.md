# EET Validator Audit — V1.1.2 + V1.1.3

Generated by the `tools/audit_eet_*.py` scripts as part of the EET review.
Scope: confirm what the validator currently enforces, surface candidate
rules not yet wired, and document spec-level findings that don't translate
into code changes.

## 1. Country-code review — N/A

The EET spec contains **zero** ISO-3166-coded fields across both V1.1.2 and
V1.1.3. The `IsoData` country-code list (with the supranational extensions
`XA` / `EU` / `XK` added for TPT V7) is therefore not exercised by EET. No
code change for this review item.

Verification: `python3 tools/audit_eet_codifications.py` — no row whose
codification mentions `ISO 3166`, `country`, or a comparable token.

## 2. Codification coverage

41 distinct codifications in V1.1.2, 42 in V1.1.3 (the only delta is the
version literal). The dominant codifications are `floating decimal`,
`Y / N`, `Number with decimal value`, plus the ISO-8601 date / ISO-4217
currency / NACE / closed-list shapes the existing `FormatRule` already
covers. The long tail contains a handful of forms worth explicit test
coverage:

| codification | shape | test coverage |
|:-------------|:------|:--------------|
| `A / A;D / B;D;F` | multi-value: each option is a `;`-joined sub-set | `EetCodificationCoverageTest` |
| `R;C / R;C;O;A` | multi-value: each option is a `;`-joined sub-set | `EetCodificationCoverageTest` |
| `EN;FR;DE` | semicolon-joined multi-language list | `EetCodificationCoverageTest` |
| `MM;IQ`, `MM;IQ;GE;FR` | semicolon-joined country-of-publication list | `EetCodificationCoverageTest` |
| `Y / Neutral` | binary with non-standard "Neutral" branch | `EetCodificationCoverageTest` |
| `Y /  N` (double space) | whitespace-noise variant of `Y / N` | `EetCodificationCoverageTest` |
| `O ; B ; E ; P ; Q ; R ; S ; T ; U ; V ; N` (V1.1.3) | semicolon-separated single-pick list | `EetCodificationCoverageTest` |
| `O ; B ; E ; P ; Q ; R ; S ; T ; U ; V` (V1.1.2 lacks `; N`) | minor closed-list drift between versions | `EetCodificationCoverageTest` |

For the full counts run `python3 tools/audit_eet_codifications.py`.

## 3. Conditional candidates not yet wired

The 6 existing + 3 new (this branch) conditional rules cover a fraction of
spec-text conditionals. The audit script `audit_eet_conditionals.py` lists
~75 NUMs whose comment explicitly mentions a conditional ("conditional to",
"if field", "when ... set to", etc.) but whose target NUM is not yet wired
into any cross-field rule.

Notable groups:

- **PAI value fields** (NUMs 105, 109, 113, ..., 237): paired with the
  PAI Y/N indicators wired in this branch. Each "Value" field carries
  "Conditional to Coverage > 0%". Wiring these requires a SFDR/RTS SME to
  define the "Coverage" field reference and the > 0 predicate semantics.
  **DEFERRED**.
- **Article-9 Paris/decarbonisation fields** (NUMs 80, 81): "Conditional
  to product being art 9". Mechanically wireable as `field 27 = "9" → 80,
  81 must be present`. SME should confirm severity (probably WARNING; the
  spec also says "Could be fulfilled for art 8").
- **Structured Product fields** (NUMs 583-588): "Conditional to being a
  Structured Product". Requires identifying the source field that flags
  Structured Product status — not obvious from the spec alone. **DEFERRED**.
- **EU Taxonomy Fossil-Gas / Nuclear fields** (NUMs 589-614): chains of
  dependencies (90000 → 90010, 90020; 90030 → 90040, 90050; …). Mechanically
  wireable but would add ~25 ConditionalRequirements. SME triage needed
  before wiring (the spec also says "Conditional to product type in field
  20040 or 20050" without specifying which value).
- **List-of-countries fields** (NUMs 615, 616): conditional to
  numeric-comparison triggers (`31210 > 0`, `31240 > 0`). Current
  `FieldPredicate` does not support numeric comparisons; would require a
  new predicate variant. **DEFERRED** until use-case demand exists.

For the full candidate list run `python3 tools/audit_eet_conditionals.py`.

## 4. M-flag drift V1.1.2 → V1.1.3

Minimal: the only profile that changed M/C flags between versions is
`SFDR_PRECONTRACT`, which gained two new C-flagged fields:
- NUM 35 (`20120_Financial_Instrument_Link_To_PCDFP_For_MOP`)
- NUM 36 (`20130_Financial_Instrument_Production_Date_PCDFP`)

Both are conditional in V1.1.3 with a comment that begins
"Conditional to 20040 or 20050 set to 8 or 9". They appear in the
Conditional-Candidates list above and are also DEFERRED until SME triage.

For the full diff run `python3 tools/audit_eet_m_flag_drift.py`.

## 5. Field-NUM stability (external-validation refs)

`EetTemplate.EXTERNAL_VALIDATION` references NUMs 3, 12, 13, 23, 24. These
NUMs are stable across V1.1.2 and V1.1.3 — confirmed by
`EetExternalValidationFieldNumStabilityTest`. The data-name paths
(`00030_EET_Producer_LEI`, `10010_…`, `10020_Manufacturer_Code`,
`20000_…_Identifying_Data`, `20010_…_Type_Of_Identification_Code`) are
identical in both versions.

PAI block NUMs are also identical between V1.1.2 and V1.1.3 (27 fields
each, NUMs 103, 104, 106, 110, 114, 118, 122, 126, 130, 134, 138, 142,
146, 150, 154, 158, 162, 166, 170, 174, 178, 182, 186, 190, 194, 198, 202).
A single shared constant in `EetRuleSet` covers both versions.
